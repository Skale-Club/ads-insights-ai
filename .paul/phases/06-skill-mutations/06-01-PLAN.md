<objective>
## Goal
Install the `google-ads-manager` skill and implement the 3 missing mutation actions (`adjustBid`, `updateCampaignBudget`, `createBudget`) in `google-ads-mutate` so Gemini tool calls actually execute against the Google Ads API instead of being silently ignored.

## Purpose
`analyze-ads` declares 7 Gemini tools but `google-ads-mutate` only handles 2 (`updateCampaignStatus`, `addNegativeKeywords`). When Gemini calls `adjustBid` or `updateCampaignBudget`, the action is currently a no-op. Closing this gap means every tool Gemini proposes can be executed for real — completing the loop between AI recommendation and campaign change.

## Output
- `google-ads-manager` skill installed in project
- `google-ads-mutate` handles 5 actions: existing 2 + `adjustBid` + `updateCampaignBudget` + `createBudget`
- All new actions validated with Zod, return correct HTTP status codes
</objective>

<task type="auto">
  <name>Task 2: Implement adjustBid, updateCampaignBudget, createBudget in google-ads-mutate</name>
  <files>supabase/functions/google-ads-mutate/index.ts</files>
  <action>
    The file already uses Zod for request validation. Extend the MutateRequestSchema enum to include the 3 new actions, then add 3 new `if (action === "...")` blocks after the existing `addNegativeKeywords` block.

    **Update MutateRequestSchema** — change the action enum from:
    ```ts
    action: z.enum(["updateCampaignStatus", "addNegativeKeywords"], ...)
    ```
    to:
    ```ts
    action: z.enum(["updateCampaignStatus", "addNegativeKeywords", "adjustBid", "updateCampaignBudget", "createBudget"], ...)
    ```
    Also add optional fields to the schema:
    - `campaignBudgetId: z.string().optional()`
    - `newBidMicros: z.number().positive().optional()`
    - `newAmountMicros: z.number().positive().optional()`
    - `budgetName: z.string().optional()`
    - `deliveryMethod: z.enum(["STANDARD", "ACCELERATED"]).optional()`
    - `adGroupId: z.string().optional()`
    - `keywordResourceName: z.string().optional()`

    **Implement `adjustBid`:**
    - Requires at least one of: `campaignId`, `adGroupId`, `keywordResourceName`
    - Requires `newBidMicros` (positive integer)
    - If `keywordResourceName`: PATCH `adGroupCriteria:mutate` with `cpcBidMicros`
    - If `adGroupId`: PATCH `adGroups:mutate` with `cpcBidMicros`  
    - If `campaignId` only: PATCH `campaigns:mutate` with `manualCpcBidMicros` (note: only works on Manual CPC campaigns)
    - Use `updateMask: { paths: ["cpc_bid_micros"] }` (or `manual_cpc_bid_micros` for campaign level)
    - Return `{ success: true, result }` or HTTP error with message

    **Implement `updateCampaignBudget`:**
    - Requires `campaignBudgetId` (the budget resource name or ID) and `newAmountMicros`
    - Build resource name: if input is numeric, construct `customers/{cleanCustomerId}/campaignBudgets/{id}`
    - PATCH `campaignBudgets:mutate` with `{ operations: [{ update: { resourceName, amountMicros: newAmountMicros }, updateMask: { paths: ["amount_micros"] } }] }`
    - Return `{ success: true, result }`

    **Implement `createBudget`:**
    - Requires `budgetName` and `newAmountMicros`
    - Optional `deliveryMethod` (default "STANDARD")
    - POST `campaignBudgets:mutate` with `{ operations: [{ create: { name: budgetName, amountMicros: newAmountMicros, deliveryMethod } }] }`
    - Return `{ success: true, resourceName: result.results[0].resourceName }`

    All new actions: return mutate response status on API error (not 200), return 400 if required fields missing.
    
    Avoid: `bidding_strategy` mutations — those require a different API endpoint and are out of scope.
    Avoid: changing existing `updateCampaignStatus` or `addNegativeKeywords` logic.
  </action>
  <verify>
    Read the modified file and confirm:

    - Each action has its own `if` block with API call
    - Each block returns proper HTTP status on error
  </verify>
